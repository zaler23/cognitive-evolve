# Self-bootstrap LOOP protocol

This protocol keeps project self-improvement evidence-based without making one run look final.

## Loop steps

1. Freeze the user goal as plain text and record the source snapshot hash.
2. Run a wide seed harvest with no low fixed project cap. If coverage is thin, continue harvesting or record why the provider stopped.
3. Evolve from the latest checkpoint only. Resume must not rerun the original seed phase unless the operator explicitly starts a fresh run.
4. Monitor device pressure, provider calls, checkpoint age, current round, family coverage, novelty, and loser-pool resurrection traces.
5. If the run reaches the configured perturb round and coverage is still stuck, recommend a target-perturb seed continuation from the latest checkpoint.
6. After completion, compare open strategy hypotheses from artifacts and candidate observations. Do not hardcode one named architecture as the project answer.
7. Convert confirmed run issues into source changes, close the debt ledger, run acceptance checks, then start the next loop from the updated source snapshot.

## Route discipline

For the current project loop, every real model route must resolve to the operator-selected GPT 5.5 high profile: seed, contract, policy, critique, rank, diagnosis, mutation, offspring, stop, and synthesis. Set `COGEV_LLM_MODEL` to that model id and set `COGEV_LLM_REQUIRED_MODEL` to the same value before starting. If route resolution is unknown, or a call ledger/status row reports a different real model, stop before the provider call or mark the run invalid. Fixture tests remain allowed only as tests; they are not evidence for a real self-bootstrap loop.

## Efficiency rule

Efficiency improvements may reduce duplicate work, improve batching, or expose better monitoring. They must not reduce seed breadth, candidate diversity, loser-pool resurrection, source evidence, or final-answer honesty.
